# crossplane-provider-opentofu

KCL schemas generated from the [`upbound/provider-opentofu`](https://github.com/upbound/provider-opentofu) CRDs. Used by consumer modules that need to emit `Workspace`, `ProviderConfig`, or `ClusterProviderConfig` resources against the **Crossplane v2 / namespaced** OpenTofu provider.

## API group

All schemas target the namespaced (v2-compatible) variant:

```
opentofu.m.upbound.io/v1beta1
```

The `.m.` in the group name denotes the namespaced provider build. This is the variant used in Crossplane v2 Compositions where `Workspace` lives in the consumer's namespace and `providerConfigRef.kind` is either `ProviderConfig` (namespaced) or `ClusterProviderConfig` (cluster-scoped).

## Schemas

Under `v1beta1/`:

- `Workspace` — the Terraform/OpenTofu workspace resource (`opentofum_upbound_io_v1beta1_workspace.k`)
- `ProviderConfig` — namespaced provider config (`opentofum_upbound_io_v1beta1_provider_config.k`)
- `ClusterProviderConfig` — cluster-scoped provider config (`opentofum_upbound_io_v1beta1_cluster_provider_config.k`)

Schemas are auto-generated by the repo task `create-object-module-from-crd` (Dagger `convert-crd`) from the upstream CRDs at:

```
https://github.com/upbound/provider-opentofu/tree/main/package/crds
```

Do not hand-edit generated files — regenerate instead.

## Usage

Import from the OCI artifact in a consumer `kcl.mod`:

```toml
[dependencies]
crossplane-provider-opentofu = { oci = "oci://ghcr.io/stuttgart-things/crossplane-provider-opentofu", tag = "0.1.0" }
```

Then in `main.k`:

```python
import crossplane_provider_opentofu.v1beta1 as otf

_ws = otf.Workspace {
    metadata.name = "example"
    spec = {
        providerConfigRef = {
            name = "default"
            kind = "ClusterProviderConfig"
        }
        forProvider = {
            source = "Inline"
            module = "..."
        }
    }
}
```
